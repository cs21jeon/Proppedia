Flutter로 부동산 앱 만들기, 기술 스택 선택 이야기

앱을 만들겠다고 마음먹으면

가장 먼저 부딪히는 질문이 있습니다.

​

"뭘로 만들지?"

​

저도 부동산백과를 처음 기획할 때

이 고민에 꽤 오래 머물렀거든요.

​

오늘은 그 과정에서 고민했던 것들을 정리해 보려 합니다.

개발에 관심 있는 분들께 조금이나마 참고가 되면 좋겠네요.

​


왜 크로스플랫폼이어야 했나

처음에는 안드로이드 앱만 생각했습니다.

​

근데 혼자 개발하잖아요.

안드로이드 따로, iOS 따로, 웹 따로 만들면 몸이 세 개라도 모자랍니다.

​

하나의 코드로 안드로이드, iOS, 웹을 동시에 커버할 수 있다면

그게 1인 개발자에게는 거의 유일한 선택지더라고요.

​

실제로 부동산백과는 하나의 코드베이스로

안드로이드, iOS, 웹, 윈도우까지 빌드하고 있습니다.

플랫폼별로 따로 개발했다면 아마 아직도 안드로이드 하나만 만들고 있었을 겁니다.

​


그래서 왜 Flutter였나

크로스플랫폼 프레임워크는 여러 가지가 있습니다.

React Native도 있고, .NET MAUI도 있고, Kotlin Multiplatform도 있어요.

​

Flutter를 고른 이유는 크게 세 가지였습니다.

​

첫 번째는 성능이었습니다.

Flutter는 네이티브로 컴파일되기 때문에 실제로 써보면 반응이 빠릅니다.

부동산 데이터를 지도 위에 올리고 스크롤 하는 동작이 많은 앱이라

이 부분이 꽤 중요했거든요.

​

두 번째는 UI 자유도였습니다.

Flutter는 위젯 단위로 화면을 구성하는데

원하는 디자인을 거의 제약 없이 만들 수 있습니다.

건축물대장 정보를 카드 형태로 보여주거나

공시지가 정보를 보기 좋게 표현할 때 이 자유도가 많이 도움이 됐습니다.

​

세 번째는 커뮤니티와 생태계였습니다.

Flutter가 나온 지 꽤 됐기 때문에 패키지도 많고

문제가 생겼을 때 검색하면 대부분 답이 나옵니다.

혼자 개발하면 검색이 곧 동료거든요.

​


나머지 기술 스택은 어떻게 골랐나

프레임워크만 정하면 끝이 아니라

상태 관리, 네트워크, 로컬 저장소, 라우팅까지 다 골라야 합니다.

​

상태 관리는 Riverpod를 선택했습니다.

Flutter에서 상태 관리 라이브러리가 여러 개 있는데

Riverpod가 코드 생성을 지원해서 보일러플레이트가 적고

테스트하기도 편했습니다.

처음에는 Provider를 쓰다가 중간에 Riverpod로 바꿨는데

바꾸길 잘했다고 생각합니다.

​

API 통신은 Retrofit을 씁니다.

인터페이스만 정의하면 나머지 코드가 자동 생성되거든요.

공공데이터 API를 여러 개 연동해야 하는 앱이라

이 자동 생성이 시간을 정말 많이 아껴줬습니다.

​

로컬 데이터베이스는 Isar를 선택했습니다.

즐겨찾기나 검색 이력 같은 데이터를 기기에 저장해야 하는데

Isar가 Flutter와 궁합이 좋고 속도도 빠릅니다.

​

화면 이동은 go_router로 처리합니다.

딥링크 지원이 잘 되고 웹에서도 URL 기반 라우팅이 가능해서

웹 버전까지 고려하면 go_router가 맞더라고요.

​

데이터 클래스는 Freezed를 써서 불변 객체로 만듭니다.

이게 없었으면 모델 클래스 하나 만들 때마다

equals, hashCode, copyWith, toJson 같은 걸 직접 써야 했을 텐데

Freezed가 다 생성해 줍니다.

​


솔직히 힘들었던 부분들

Flutter가 좋은 점만 있는 건 아닙니다.

​

빌드 시간이 꽤 깁니다.

코드 생성기를 여러 개 쓰다 보니

build_runner 돌리면 한참 기다려야 해요.

코드 한 줄 고치고 결과 보려고 1분씩 기다리는 건 좀 고역이었습니다.

​

플랫폼별로 미묘하게 다르게 동작하는 부분도 있습니다.

안드로이드에서 잘 되던 게 웹에서는 안 되거나

iOS에서 권한 처리가 다르거나 하는 경우가 종종 있거든요.

하나의 코드라고 해도 플랫폼 분기 처리는 피할 수 없었습니다.

​

카카오맵 같은 네이티브 SDK를 Flutter에서 쓰려면

플러그인에 의존해야 하는데

공식 지원이 아니다 보니 업데이트가 늦거나 버그가 있을 때 난감합니다.

​


다시 시작한다면 뭘 바꿀까

아키텍처를 처음부터 Clean Architecture로 잡을 겁니다.

저는 처음에 대충 짰다가 나중에 전부 리팩터링했거든요.

규모가 커지니까 코드가 엉켜서 손대기 무서워지는 시점이 오더라고요.

​

테스트 코드도 처음부터 작성할 겁니다.

"나중에 쓰지" 하다가 결국 안 쓰게 되는 게 테스트 코드인데

지금 와서 보면 테스트가 있었으면 버그를 훨씬 빨리 잡았을 것 같습니다.

​

그리고 CI/CD 파이프라인도 초기에 세팅했을 겁니다.

매번 수동으로 빌드하고 배포하는 게 시간이 꽤 들거든요.

​


기술 스택은 결국 도구일 뿐이라고 생각합니다.

어떤 기술이 최고인가보다

내 상황에 맞는 기술이 무엇인가가 더 중요했습니다.

​

1인 개발자이고 여러 플랫폼을 커버해야 하고

빠르게 결과물을 내야 하는 상황에서

Flutter는 저에게 꽤 잘 맞는 선택이었네요.
